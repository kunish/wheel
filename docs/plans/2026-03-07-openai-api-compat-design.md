# OpenAI API Compatibility Design

**Goal:** Make `wheel`'s public `/v1/*` surface behave like an OpenAI-compatible API for core inference and multimedia workflows, so common OpenAI SDK clients can call it directly.

## Current State

- `apps/worker/internal/handler/relay.go` already exposes OpenAI-style routes such as `/v1/models`, `/v1/chat/completions`, `/v1/embeddings`, `/v1/responses`, `/v1/images/generations`, and `/v1/audio/*`.
- `apps/worker/internal/relay/parser.go` can classify these endpoints, but request parsing is still JSON-only and assumes a simple `{model, stream}` body.
- `apps/worker/internal/relay/multimodal.go` contains JSON-based helpers for multimodal upstream calls, but the handler path does not yet treat multipart uploads and binary responses as first-class OpenAI-compatible behaviors.
- `apps/worker/internal/handler/relay_log.go` returns a basic OpenAI-style error envelope, but it does not yet centralize richer OpenAI-compatible error and header behavior.

## Scope

### In Scope

- `/v1/models`
- `/v1/chat/completions`
- `/v1/responses`
- `/v1/embeddings`
- `/v1/images/generations`
- `/v1/audio/speech`
- `/v1/audio/transcriptions`
- `/v1/audio/translations`
- `/v1/moderations`

### Out of Scope for This Phase

- `/v1/files`
- `/v1/uploads`
- `/v1/vector_stores`
- `/v1/fine_tuning/*`
- `/v1/realtime/*`
- `/v1/assistants/*`
- OpenAI-native batch resource semantics; existing `wheel` `/v1/batch` remains a product-specific extension

## Product Contract

- `wheel` keeps one public `/v1/*` entrypoint rather than exposing a second vendored OpenAI server.
- External clients see OpenAI-compatible paths, request formats, status codes, error shapes, and streaming behavior.
- Internally, requests still flow through `wheel` routing, API-key access control, channel selection, retry, plugin hooks, and observability.
- Compatibility is defined as "common OpenAI SDK/client workflows succeed against wheel" rather than byte-for-byte parity with every OpenAI resource.

## Architecture

### 1. Public Route Layer Stays in `RelayHandler`

- `apps/worker/internal/handler/relay.go` remains the authoritative registration point for `/v1/*` routes.
- We do not move responsibility into vendored `CLIProxyAPIPlus`; vendored code stays an internal upstream integration detail.
- Route registration stays explicit so `wheel` can preserve custom endpoints like `/v1/async/chat/completions` and `/v1/batch` without mixing them into the OpenAI compatibility contract.

### 2. Endpoint-Aware Request Parsing

- Request parsing is split by endpoint family before channel selection:
  - JSON requests: `chat/completions`, `responses`, `embeddings`, `moderations`, `images/generations`, `audio/speech`
  - multipart requests: `audio/transcriptions`, `audio/translations`
- The parsed request object must carry enough metadata for downstream execution:
  - request type
  - model
  - stream flag
  - raw body bytes for logging/replay
  - normalized JSON body when available
  - multipart form payload or file metadata when required
  - whether the response must be streamed as SSE or binary

### 3. Two Execution Lanes

- **Structured JSON lane** for `responses`, `chat/completions`, `embeddings`, `moderations`, and image-generation JSON responses.
- **Binary/multipart lane** for speech output and transcription/translation uploads.
- Both lanes still reuse the same channel-selection and retry loop, but they need endpoint-aware upstream request builders and response writers.

### 4. OpenAI-Compatible Response Shaping

- `/v1/models` remains generated by `wheel`, but the response shape is stabilized for OpenAI clients.
- Error responses are normalized through one OpenAI-compatible helper.
- Success responses preserve upstream OpenAI-compatible payloads whenever possible rather than reformatting them unnecessarily.
- Where upstream providers differ, `wheel` adapts to the smallest stable OpenAI-compatible subset needed by mainstream SDKs.

## Data Flow

1. Request enters `apps/worker/internal/handler/relay.go` via an explicit `/v1/*` route.
2. Endpoint-aware parsing classifies the request and extracts model plus transport requirements.
3. Existing access checks, plugin pre-hooks, routing rules, and channel selection run as they do today.
4. Retry execution chooses the correct upstream builder based on request type:
   - standard JSON proxy path for text-like requests
   - multimodal JSON proxy path for images/moderations
   - multipart upload path for transcription/translation
   - binary passthrough path for speech output
5. Response is written back in an OpenAI-compatible shape, with consistent error envelopes and selected headers.
6. Logging, observer spans, retry accounting, and plugin post-hooks still run from `wheel`.

## Compatibility Details

### Models

- Continue returning OpenAI list shape from `/v1/models`.
- Keep the field set small and stable: `object`, `data[]`, `id`, `created`, `owned_by`.
- Filter by API-key model access exactly as today.

### Chat and Responses

- Keep both `/v1/chat/completions` and `/v1/responses` first-class; do not force migration from one to the other.
- Preserve SSE behavior for stream-enabled callers.
- Preserve OpenAI-style JSON error objects for validation, routing, and upstream failures.

### Embeddings and Moderations

- Treat these as structured JSON proxy endpoints with stricter request validation and consistent response envelopes.

### Images

- First phase guarantees stable request/response handling for `POST /v1/images/generations`.
- Response compatibility targets common OpenAI JSON output conventions rather than every optional field in the newest spec.

### Audio

- `POST /v1/audio/speech` must support binary/audio responses without forcing JSON wrapping.
- `POST /v1/audio/transcriptions` and `POST /v1/audio/translations` must accept multipart uploads and normalize them to the configured upstream channel.

## Error Handling

- Standard OpenAI response envelope:

```json
{
  "error": {
    "message": "...",
    "type": "...",
    "param": null,
    "code": null
  }
}
```

- Map internal failures into stable OpenAI categories:
  - invalid request / validation failures
  - forbidden model access
  - not found model/group
  - upstream rate limiting
  - upstream server errors
- Preserve Anthropic-specific formatting only for Anthropic inbound compatibility; OpenAI routes always return OpenAI-style errors.

## Headers and Observability

- Preserve or synthesize request identifiers (`x-request-id`) for troubleshooting.
- Preserve rate-limit headers when the upstream provides them and they are safe to surface.
- Keep current `wheel` logging, tracing, retry, and plugin hook behavior.

## Testing Strategy

- Route registration tests for all supported `/v1/*` endpoints.
- Parser tests for:
  - JSON request bodies
  - missing model handling
  - multimodal defaults
  - multipart transcription/translation parsing
- Handler contract tests for:
  - `/v1/models` response shape
  - OpenAI error envelope shape
  - binary speech response passthrough
  - transcription/translation multipart uploads
- Integration tests covering at least:
  - `chat/completions`
  - `responses`
  - `embeddings`
  - `images/generations`
  - `audio/transcriptions`
  - `audio/speech`

## Non-Goals

- No attempt to fully mirror every OpenAI platform resource in one pass.
- No new persistence model for files, vector stores, or fine-tuning jobs.
- No replacement of `wheel` custom async/batch APIs with OpenAI-native resource models in this phase.
